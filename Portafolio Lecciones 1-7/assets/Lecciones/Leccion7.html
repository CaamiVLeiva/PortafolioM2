Lección 7

GIT Y GITHUB


1. Necesidad de un repositorio de código fuente:

Permite mantener un historial de cambios en el código, facilita la colaboración entre desarrolladores trabajando en paralelo de manera segura, oferece herramientas para revisar y comentar los cambios antes de integrarlos lo cual mejora la calidad, organización y seguridad.

2. Instalación, configuración y comandos básicos:
  • Para descargar Git debemos ir a la página oficial https://git-scm.com/.
  • Una vez descargado abrimos "Git Bash" en el cual ingresaremos los siguientes comandos para configurar nombre y correo electrónico: 'git config --global user.name "Tu Nombre"' y 'git config --global user.email "tu@email.com"' Con el comando 'git config --list' podemos visualizar si se guardaron los cambios.
  • Algunos comandos básicos son:
    • git init: Se utiliza para iniciar un nuevo repositorio Git en un directorio existente.
      • 
                        git init
    • git add 'nombre_archivo': Se utiliza para agregar archivos al área de preparación.
      • 
                        git add . #Agrega  todos los archivos al area de preparación.
    • 'git commit -m"Comentario"': Se utiliza para guardar los cambios en el repositorio local junto con un mensaje descriptivo.
      • 
                        git commit -m "Mensaje descriptivo del cambio." #Realiza el primer envío a la base de datos local.
    • 'git status': Verifica el estado del repositorio para ver qué archivos han sido modificados, agregados o eliminados.
      • 
                        git status
    • 'git remote': Al crear un repositorio en GitHub puedes configurar tu repositorio local para apuntal al remoto.
      • 
                        git remote add origin URL_del_repositorio_remoto 
                        #El enlace aparece en tu repositorio GitHub
    • 'git push': Una vez hechos los cambios necesarios, se utiliza este comando para enviarlos al repositorio remoto.
      • 
                        git push -u origin main
3. Commits y restauración de archivos:

  • Utilizamos el comando 'touch' para crear un archivo en la ruta actual
  • Usamos los comandos 'git add' y 'git commit -m' para subir el archivo al area de preparación y para guardar los cambios.
  • Realizamos un cambio en archivo, para volver a subirlo y guardarlo.
  • Repetimos el paso anterior pero agregando mas texto al archivo.
  • Ahora, con los cambios realizados y dos commits hechos ingresamos el comando 'git checkout HEAD-1 NOMBRE_ARCHIVO'. Lo que nos permite restaurar una versión anterior del archivo, en este caso -1(una version anterior).

4. Cambios de nombres:

Para cambiar el nombre de un archivo usamos el comando 'git mv nombre_antiguo nombre_nuevo'. Con el comando 'ls' verificamos el nombre del archivo en el directorio.

5. Ignorando archivos:

  • Para ignorar un archivo o una carpeta se utiliza el archivo ".gitignore".
  • Se debe añadir el contenido que deseas ignorar.
  • Guardamos y verificamos que el archivo que deseamos ignorar no se suba a nuestro repositorio.

6. Ramas uniones, conflictos y tags:

  • Utilizamos el comando 'git branch nombre_rama' para crear una nueva rama
  • Para cambiar a la rama creada utilizamos el comando 'git checkout nombre_rama'
  • Para fusionar una rama con la rama actual utilizamos el comando 'git merge nombre_rama'
  • Si hay conflictos entre las ramas, Git nos avisará y mostrará los conflictos.
    • Problema: Al intentar cambiar de la rama rama_prueba a la rama main, Git indicó que había cambios no comprometidos en el archivo Portafolio Lecciones 1-7/assets/Lecciones/Leccion7.html. Al fusionar las ramas rama_prueba y main, Git mostró archivos no rastreados y cambios en otros archivos.
    • Resolución: Se realizó un commit de los cambios en la rama rama_prueba. Se cambiaron a la rama main. Se fusionaron las ramas rama_prueba y main. Se revisaron los archivos no rastreados y se agregaron al repositorio según fuera necesario. Se verificó que los cambios estuvieran comprometidos correctamente y se aseguró de que no hubiera conflictos pendientes.
  • Para etiquetar una versión específica del código utilizamos el comando 'git tal nombre_tag'

7. Stash y Rebase:

Usamos el comando 'git stash' para guardar temporalmente cambios no comprometidos. 

                
                    git stash  save "Mensaje descriptivo del cambio"
Usamos el comando 'git rebase -i HEAD~n' 
                
                    git rebase -i HEAD~1 #  rebasing el ultimo commit.

8. Repositorios remotos, Push y Pull:

  1. Creamos un repositorio en GitHub: Mi link
  2. Copiamos la URL del repositorio que hemos creado. Agregamos este repositorio como remoto con el siguiente comando: 
                            
                            git remote add origin urlDelRepositorio
    Verificamos los remotos agregados con el comando: 
                            
                            git remote show nombreRemoto
  3. Con el comando 'git pull origin nombre_rama' obtenemos los cambios del repositorio remoto en tu repositorio local
  4. Con el comando 'git push origin nombre_rama' Hacemos un push a nuestro repositorio remoto.
  5. Podemos realizar otro pull o push para verificar que no quede nada en preparación.

9. Fetch v/s Pull

Tabla Comparativa

git fetch git pull
Función Descarga cambios del repositorio remoto sin fusionarlos Descarga y fusiona automáticamente cambios remotos
Acciones realizadas Solo descarga cambios Descarga y fusiona cambios
Actualización local No actualiza automáticamente la rama local Actualiza automáticamente la rama local con los cambios
Control de fusiones Requiere fusionar cambios manualmente Fusiona automáticamente los cambios
Ventajas Permite revisar los cambios antes de fusionarlos Actualiza rápidamente la rama local con los cambios
Desventajas Requiere pasos adicionales para fusionar cambios Menos control sobre los cambios antes de la fusión
En temas de comando tambien se parecen bastante: 
            
                $git fetch origin //Obtiene los cambios del repositorio remoto y los guarda en tu repositorio local pero no se fusionan automaticamente en tu rama local
            
        
            
                $git pull origin main //Primero ejecuta git fetch para obtener los cambios del repositorio remoto, para luego fusionarlos automaticamente

13. Flujos de trabajo con Github:

Modelo de Trabajo Características Ventajas
Modelo de Bifurcación Cada desarrollador trabaja en su propia bifurcación del repositorio principal. Permite un enfoque descentralizado. Los desarrolladores pueden trabajar independientemente sin afectar el repositorio principal.
Modelo de Rama por Característica Se crea una nueva rama para cada nueva característica o cambio en el código. Facilita el seguimiento de las características implementadas. Permite una gestión más granular de los cambios en el código.
